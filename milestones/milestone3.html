Our system autonomously explores unknown environments using a TurtleBot4 Lite ground robot, simultaneously building a 2D occupancy map via SLAM and detecting semantic hazards (HAZMAT signs, fire, cliffs, potholes) using a multi-layer sensor fusion pipeline. The robot selects and navigates to frontier cells — boundaries between known-free and unknown space — using a nearest-first wavefront strategy with A* path planning on a dynamically-built costmap.

2. Algorithm

The system comprises four tightly-coupled custom ROS 2 nodes. Each implements a distinct algorithmic layer in the autonomy stack.

2.1 Frontier Explorer Node

The frontier explorer subscribes to the SLAM occupancy grid and identifies frontier cells — free cells adjacent to unknown cells. Clusters are formed using connected-component labeling and scored by distance from the robot using a nearest-first wavefront metric:

$$\text{score}(c) = \frac{1}{d(r, c) + \varepsilon}$$
Eq. 1 — Frontier scoring (wavefront / nearest-first)

where $d(r, c)$ is the Euclidean distance from robot position $r$ to frontier centroid $c$, and $\varepsilon = 10^{-6}$ prevents division by zero. Clusters with fewer than $N_{\min} = 10$ cells are discarded as noise.

$$\mathcal{F} = \left\{ c \in \text{Free} \;\middle|\; \exists\, n \in \mathcal{N}(c) : n \in \text{Unknown} \;\land\; \mathcal{N}(c) \cap \text{Occupied} = \emptyset \right\}$$
Eq. 2 — Frontier cell definition
# frontier_explorer_node.py — core scoring (line 94)
score = 1.0 / (dist + epsilon)  # nearest frontier first (wavefront)

The LiDAR sector analysis accounts for the +90° mounting offset confirmed in the TurtleBot4 Lite URDF (rpy="0 0 π/2"):

$$\theta_{\text{robot}} = \theta_{\text{LiDAR}} - \frac{\pi}{2}$$
Eq. 3 — LiDAR angle correction for RPLIDAR A1M8 mounting offset
$$d_{\text{front}} = \min_{i \in [-120°, -60°]} r_i, \quad d_{\text{left}} = \min_{i \in [-30°, 30°]} r_i, \quad d_{\text{right}} = \min_{i \in [150°, 210°]} r_i$$
Eq. 4 — Sector minimum distances (robot frame)

HAZMAT inference uses DeepHAZMAT's YOLOv3-tiny architecture (13 classes, 7M parameters, 15.8 GFLOPs), running at ~30ms per frame on an RTX 5070 Ti GPU with CUDA 13.1.

2.3 Navigation Planner Node

The navigation planner implements a custom /navigate_to_pose action server replacing Nav2. It builds a 2D costmap from the SLAM OccupancyGrid and plans paths using A*:

$$f(n) = g(n) + h(n), \quad h(n) = \|n - \text{goal}\|_2$$
Eq. 5 — A* heuristic (Euclidean distance)
$$C(n) = \begin{cases} 0 & d_{\text{wall}}(n) > r_{\text{soft}} \\ 10 & r_{\text{hard}} < d_{\text{wall}}(n) \leq r_{\text{soft}} \\ 80 & 0 < d_{\text{wall}}(n) \leq r_{\text{hard}} \\ 1000 & d_{\text{wall}}(n) = 0 \end{cases}$$
Eq. 6 — Costmap cell cost tiers ($r_{\text{hard}}=0.15m$, $r_{\text{soft}}=0.18m$)

A proportional controller drives the robot along pruned waypoints:

$$v = \text{clip}(K_P^{lin} \cdot d_{wp},\; V_{min},\; V_{max}), \quad \omega = \text{clip}(K_P^{ang} \cdot e_\psi,\; -\omega_{max},\; \omega_{max})$$
Eq. 7 — Proportional velocity controller ($K_P^{lin}=0.80$, $K_P^{ang}=0.40$, $V_{max}=0.35$ m/s)

2.4 Behavior Coordinator Node

The coordinator implements a 6-state finite state machine managing the exploration lifecycle:

$$\mathcal{S} = \{\text{IDLE}, \text{SELECTING}, \text{PENDING}, \text{NAVIGATING}, \text{ARRIVED}, \text{DONE}\}$$
Eq. 8 — State space

State Transitions

FromToCondition
SELECTINGPENDINGFrontier available, navigator server ready
PENDINGNAVIGATINGGoal accepted by action server
NAVIGATINGSELECTINGGoal succeeded or frontier consumed > $r_{lock}=3.0m$
NAVIGATINGSELECTINGGoal failed, failure count < $N_{max}=5$
NAVIGATINGDONEBattery < 15% or E-stop triggered
SELECTINGDONEAll frontiers visited/blacklisted

Frontier stability is enforced via a proximity lock: if the robot is within $r_{lock} = 3.0$m of its current goal anchor, frontier updates are ignored — preventing oscillation between competing frontiers as the map updates.

4. Ethical Impact Statement

Autonomous mobile robots operating in shared human environments raise important ethical considerations that must be addressed both in current implementation and future iterations of this system.

🔒

Privacy

The OAK-D camera continuously captures RGB and depth imagery during exploration. In current lab testing this data is processed locally and not stored persistently. However, deployment in public or workplace environments would necessitate explicit consent protocols, automatic face/person blurring (via YOLO person detection with anonymization), and a data retention policy compliant with FERPA and relevant privacy regulations. The system currently logs all hazard detections to ROS bags — these logs must be treated as sensitive data if they contain identifiable imagery.

⚡

Safety

The TurtleBot4 Lite weighs approximately 3.3kg and operates at up to 0.35 m/s, producing kinetic energy of ~0.2J — sufficient to cause discomfort on human contact. Our system implements a multi-layer safety architecture: hardware cliff sensors trigger immediate E-stop, the navigation planner inflates walls by 0.15-0.18m to maintain safe clearance, and the CRITICAL_HAZARDS set ({CLIFF, POTHOLE}) triggers exploration halt. Future iterations should add human detection via the depth camera to reduce speed or stop when people are within 1m, and implement a physical emergency stop accessible to bystanders.

⚖️

Bias & Limitations

The RPLIDAR A1M8 cannot detect glass walls, transparent barriers, or highly reflective surfaces — creating systematic blind spots. The DeepHAZMAT model was trained on standardized placard images and may underperform on damaged, faded, or non-standard HAZMAT signs used outside North America. The YOLOv5 fire model was trained on video fire imagery and shows reduced performance on printed/displayed representations. From a Justice perspective, deployment in under-resourced facilities may face sensor limitations (glass walls, poor lighting) not present in standard lab settings, creating unequal safety outcomes. From a Utilitarian perspective, the system's exploratory benefit in hazardous environments (search and rescue, industrial inspection) outweighs current limitations, provided the limitations are clearly communicated to operators.

Ethical Framework Application

Utilitarian: The system maximizes utility in scenarios where human entry would be dangerous (post-disaster search, industrial hazmat surveys). The expected benefit to rescuers and inspectors outweighs the risk of false negatives in hazard detection, provided humans remain in supervisory roles.

Justice: Ensuring equitable safety outcomes requires the system to perform reliably across diverse environments, not only well-lit, obstacle-free lab corridors. Future work must include testing in varied conditions (low light, cluttered environments, outdoor settings) to identify and mitigate performance disparities.

Virtue Ethics: Engineers deploying this system bear responsibility for honest communication of its limitations to end users, regular revalidation as deployment contexts evolve, and ensuring meaningful human oversight is maintained throughout autonomous operation.
